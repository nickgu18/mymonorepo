### Task
Implement the remaining expectations for the "Heal" feature in Bench-Hub (b/484405237) based on the original Buganizer issue. 
Specifically:
1. Detect "broken" runs when they finish.
2. Ensure the Heal button only shows up for identified "broken" runs.
3. Immediate task status updates on the UI: errored tasks go back to the running state when "Heal" is clicked.
4. Filter out "heal jobs" from the run history page by default.
5. Re-evaluate original run when the heal job finishes (clear broken status or keep Heal button).

### analysis
The current prototype implemented in PR #57 (Comment #8) provides the core capability to retry failed tasks by submitting a new Cloud Batch job. However, it lacks robust state management:
* **Status Detection:** The frontend dynamically checks `!isRunning && hasErrors` instead of relying on a concrete `BROKEN` status from the backend.
* **Immediate UI Update:** The `POST /api/jobs/{job_id}/heal` endpoint submits a batch job but doesn't transition the existing BigQuery task records from `ERROR` to `QUEUED`/`STARTING`. Thus, the UI continues showing `ERROR` until workers pick up the tasks.
* **Filtering Heal Jobs:** Heal jobs are tagged but not explicitly filtered out of the `/api/jobs` list for the run history dashboard.
* **Re-evaluation:** There is no loop to check when a heal job finishes to update the parent job's status.

### fix_plan
To start simple (Purpose Driven Development), we will break this down and implement it incrementally:

**Step 1: Heal Job Filtering (Quick Win)**
* Update the backend `/api/jobs` endpoint or BigQuery query to filter out jobs with the `heal_of` tag by default. 
* Update the `RunBrowser` in the web-ui to handle this filtering if done purely on the frontend, but backend filtering is preferred.

**Step 2: Explicit "BROKEN" Status & UI Alignment**
* Update job status transitions. When a job completes, if there are failed tasks, transition its status to `BROKEN` (or a similar state) rather than `COMPLETED`.
* Update `RunSummary.tsx` to only show the "Heal" button when `status === 'BROKEN'`.

**Step 3: Immediate Task State Reversal**
* In `orchestrator/main.py` -> `heal_job`, right after submitting the batch jobs, update the status of the `tasks_to_heal` in BigQuery to `QUEUED`.
* Ensure the frontend receives this update and displays the tasks as Queued/Running immediately.

**Step 4: Post-Heal Re-evaluation**
* Modify the worker/orchestrator completion logic. When a job finishes, check if it has a `heal_of` tag. If so, trigger a re-evaluation of the parent job (check if the parent job's tasks are now 100% successful). If so, change parent job status to `COMPLETED`.
