Founder Ranking Pipeline

Iteration: 4 (Proposed)
Key Improvement: This iteration fixes the temporal logic by using the best exit (max multiple) company as the label instead of the last labeled company. Each founder now has a single snapshot anchored at label_company_founded_year, features are built only from information available up to that year, and post-split performance/network/team features are fit on Train only and applied to Val/Test, ensuring proper time-respecting, leak-free predictions.

High-level Flow

Phase 1: Environment Setup
Same as Iteration 3
  • Load configuration via load_config().
  • Set up MLflow tracking and logging.
  • Initialize random seeds for reproducibility.
↓
Phase 2: Data Loading & Feature Engineering CHANGED
Explicit label company (max multiple) and pre-split founder snapshot
  • 2.1 Data Loading & Cleaning: Load raw training/ranking tables and targets with load_raw, clean, and load_targets (same as earlier iterations).
  • 2.2 Label Company Selection (max multiple):
    • For each founder, identify all founded companies that appear in target_variable_training.
    • Select the company with the highest multiple (max_founder_multiple).
    • Set per founder: label_max_founder_multiple, label_company_id, label_company_founded_year, label_company_industry.
    • This label construction happens before any train/val/test split.
  • 2.3 Initial Founder Snapshot (Pre-split matrix):
    • Use FeaturePipeline to build local, time-respecting features per founder (e.g., performance of the label company, education_tier, education_level_score).
    • Override metadata so that company_id, industry, and company_founded refer to the label company.
    • Attach the raw label label_max_founder_multiple and label metadata (label_company_id, label_company_founded_year, label_company_industry).
    • Result: one row per founder with only local, non–split-aware features plus the raw label and label metadata; this is the matrix that will be split.
↓
Phase 3: Data Splitting
Temporal train/val/test split by label_company_founded_year
  • Sort the initial founder matrix by label_company_founded_year within each industry.
  • Split into 70% Train / 15% Validation / 15% Test (oldest → newest).
  • Labels (label_max_founder_multiple) are already attached—fine because they're future outcomes relative to the snapshot and were computed per founder from targets only.
  • Return: X_train, y_train, X_val, y_val, X_test, y_test + groups.
↓
Phase 4: Data Validation
Same validation checks as Iteration 3
  • Print per-industry counts and label distributions for all splits.
  • Verify temporal ordering (no train rows after test rows).
  • Check for missing values and feature ranges.
↓
Phase 4.5: Advanced Preprocessing (Optional) NEW
Post-split, train-fitted feature transformations
  • Performance features (train-fitted):
    • From train founders' histories, compute: founder_has_perf, founder_perf_mean, founder_perf_max, founder_perf_last.
    • Use a cut_off_date (typically the founder’s label_company_founded_year) so performance history after that date is ignored.
    • Join these aggregates into Train / Val / Test by person_id, using train-derived mappings for Val/Test.
  • Network features (train-fitted):
    • Build a co-worker graph from train experience and derive: network_quality_1st, network_size_1st.
    • Apply the same cut_off_date when building the graph so edges reflect only relationships active up to label_company_founded_year.
    • Apply these network scores to Val/Test founders without recomputing the graph.
  • Team features (train-fitted):
    • Compute team_size per label company from train founders' histories, counting only founders whose roles start on or before the relevant cut_off_date.
    • Attach the same team_size mapping to founders in Val/Test whose label company matches.
  • Discipline: all such transformations are:
    • Fit on Train only (aggregates, graphs, buckets).
    • Applied to Val/Test using train-derived parameters, avoiding leakage.
↓
Phase 5: Model Training
Same train_ranker approach as Iteration 3
  • Train XGBRanker with Train data, using Validation as eval set.
  • MLflow logs parameters, label histograms, feature importance, NDCG@20.
  • Validation used for tuning; Test remains held-out.
↓
Phase 6: Model Persistence
Same as Iteration 3
  • Save trained model and feature metadata to mlruns/models/notebook_runs/<timestamp>.
  • Persist model bundle for later loading.
↓
Phase 7: Inference
Same as Iteration 3
  • Reload model bundle with load_model_bundle.
  • Build ranking features via FeaturePipeline on clean_ranking.
  • Run predict_batch to get scores and rankings.
↓
Phase 8: Ranking Explanation
Same SHAP analysis as Iteration 3
  • Generate global and per-industry SHAP plots.
  • Compute top-N SHAP explanations for interpretability.
Summary of Changes: Iteration 4 defines the training unit as a single founder snapshot at label_company_founded_year for their best exit (max multiple) company. It builds a pre-split founder matrix with only local, time-respecting features plus the raw label label_max_founder_multiple, then performs a temporal 70/15/15 split by industry and label_company_founded_year. Finally, performance, network, and team features are added after the split, fit on Train only and applied to Val/Test, preventing subtle information leakage while keeping the pipeline aligned with a realistic, time-aware production scenario.
