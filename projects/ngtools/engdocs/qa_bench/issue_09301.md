### Task

`agent-prototypes/agent_prototypes/scripts/qabench_pipeline.py` is running into quota issue with gemini cli.
The gemini cli is invoked via `agent-prototypes/agent_prototypes/utils/gemini_cli_runner.py` which then calls `agent-prototypes/agent_prototypes/utils/utils.py` with  run_shell_command_async.

Now the problem I face originates from one key being used too many times...

What I need is a key rotator, that we can

1. configure how many concurrent job this key accepts
2. swaps for the next key if one key is filled up

This should be placed under `agent-prototypes/agent_prototypes/utils`, i.e gemini_key_rotator.py

The keys should be stored in a single config file for now (gemini_key_config.yaml)
This file should not be checked in.

### analysis

The `qabench_pipeline.py` script utilizes `asyncio` to process multiple questions and generate multiple answers for each question concurrently. This results in a significant number of parallel processes making calls to the Gemini CLI. Since the current implementation relies on a single Gemini API key, it quickly hits the rate limits and quota restrictions for that key, causing the pipeline to fail. The core issue is the absence of a mechanism to distribute the API requests across a pool of available keys.

### fix_plan

1.  **Create `gemini_key_config.yaml`**:
    *   A new file at `agent-prototypes/gemini_key_config.yaml` to store a list of Gemini API keys and their concurrency limits.
    *   Example:
        ```yaml
        keys:
          - key: "AIza..."
            concurrency: 2
          - key: "AIzb..."
            concurrency: 2
        ```

2.  **Update `.gitignore`**:
    *   Add `gemini_key_config.yaml` to `agent-prototypes/.gitignore` to prevent committing keys.

3.  **Implement `GeminiKeyRotator`**:
    *   Create `agent-prototypes/agent_prototypes/utils/gemini_key_rotator.py`.
    *   Implement a thread-safe, `asyncio`-compatible singleton class `GeminiKeyRotator`.
    *   The class will load keys from the YAML file, track the usage of each key, and use an `asyncio.Condition` to manage acquiring and releasing keys, waiting when no keys are available.
    *   It will expose an `async` context manager for acquiring/releasing keys.

4.  **Integrate Rotator into `gemini_cli_runner.py`**:
    *   In `run_gemini_cli_async`, use the `GeminiKeyRotator`'s context manager (`async with`) to acquire a key.
    *   Pass the acquired key to the subprocess by providing a modified environment dictionary to the `asyncio.create_subprocess_shell` call, ensuring the key is scoped only to that specific execution.
