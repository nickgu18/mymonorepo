# Analysis of b/484405237: "Heal" Feature Expectations vs. Original Plan

## Overview
This document compares the clear expectations for the "Heal" feature outlined in the description of Buganizer issue b/484405237 against the original prototype plan documented in Comment #8 (from PR #57). 

## Comparison and Additional Steps

Based on the investigation of the `projects/bench-hub` codebase and the latest issue description, the following gaps have been identified between the prototype (original plan) and the required final implementation:

### 1. Automatic "Broken" Detection & Status Management
* **Expectation:** "When a run finishes, automatically detect whether it's broken / needs healing."
* **Current Plan/Implementation:** The system computes whether a run can be healed dynamically on the frontend (`!isRunning && hasErrors` in `RunSummary.tsx`). 
* **Additional Steps Required:** 
  * Implement explicit backend logic to evaluate the run upon completion and transition its status to a persisted "BROKEN" or "NEEDS_HEAL" state.
  * The "Heal" button should conditionally render based on this explicit status rather than just counting errors on the fly.

### 2. Immediate Task State Reversal on UI
* **Expectation:** "When I click Heal, I should be able to see the errored tasks go back to the running state on the original run's summary page."
* **Current Plan/Implementation:** The `POST /api/jobs/{job_id}/heal` endpoint submits a new Cloud Batch job and returns. The UI shows a "Healing X tasks..." notification but the task list still shows them as `ERROR` until the new worker sidecars start and overwrite the status in BigQuery.
* **Additional Steps Required:**
  * When a heal is triggered, the backend must explicitly update the status of the selected failed tasks in BigQuery back to `QUEUED` or `STARTING`.
  * The frontend `RunSummary` must reflect this optimistic state immediately upon receiving the successful heal trigger response.

### 3. Heal Job Association & History Filtering
* **Expectation:** "If the Heal operation kicks off one or more new jobs, those should be marked as 'heal jobs' and associated with the original run id. Heal jobs should be hidden / filtered out of the run history page by default."
* **Current Plan/Implementation:** The orchestrator creates new `batch_job_id`s with `-heal-` in the name and tags them `heal_of={run_id}`. However, they are still submitted as standard jobs.
* **Additional Steps Required:**
  * Introduce an explicit `is_heal_job` flag or enforce the `heal_of=` tag filtering at the API/database query layer for the run history.
  * Update the frontend `RunBrowser` to explicitly hide these heal jobs by default, perhaps reusing the existing collapsed filtering panel logic.

### 4. Post-Heal Re-evaluation
* **Expectation:** "When the heal job(s) finish, the original job should be re-evaluated. If it's now fully healed, the broken status should be cleared and the Heal button should be removed from its run summary page. Otherwise, the Heal button should remain so the user can keep trying to heal until all the tasks complete without errors."
* **Current Plan/Implementation:** The original plan only focuses on firing off the heal jobs. There is no mechanism to detect when a heal job *finishes* to trigger a re-evaluation of the parent run.
* **Additional Steps Required:**
  * Add a background watcher, pub/sub trigger, or periodic job to detect when a "heal job" finishes.
  * Upon heal job completion, re-evaluate the parent run's tasks. If 100% of tasks are successful, transition the parent run to `COMPLETED` and clear the broken status. If not, transition it back to `BROKEN`.

## Conclusion
The original PR #57 provided the core mechanism for re-executing failed tasks via Cloud Batch. To meet the full product requirements, significant additions are needed around **explicit state machine transitions** (Broken -> Healing -> Re-evaluated), **immediate UI feedback** for task statuses, and **parent-child job relationship filtering** in the Run History dashboard.


## recommendation

 To start simple, here is a breakdown of the plan:


   1. Step 1: Heal Job Filtering (Quick Win) - Ensure heal jobs are hidden from the Run History page by default
      (updating backend endpoints and BigQuery filtering).
   2. Step 2: Explicit "BROKEN" Status - The system should natively transition job statuses to BROKEN if tasks
      fail upon completion, and the UI should only show the Heal button if this state is detected.
   3. Step 3: Immediate Task State Reversal - When the "Heal" button is clicked, immediately update those
      specific failed tasks back to QUEUED in BigQuery so the UI updates instantly.
   4. Step 4: Post-Heal Re-evaluation - Add a loop or completion logic to detect when a heal job finishes,
      triggering a re-evaluation of the parent run so its broken status is cleared if fully resolved.
