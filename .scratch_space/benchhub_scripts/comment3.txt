Excellent point regarding historical artifacts! You've highlighted the one area where the current design makes a tradeoff: **Artifact Overwriting in GCS.**

Currently, when the heal job completes, it explicitly overwrites the artifacts of the original run:
`cp "$WORK_DIR/raw_logs.zip" "$DEST/"` (which resolves to `gs://$GCS_BUCKET/$BENCHHUB_RUN_ID/`)

Because we use the identical `BENCHHUB_RUN_ID` as the destination, the `raw_logs.zip` from the first failed attempt is physically overwritten by the new, merged `raw_logs.zip` containing the successful healed tasks.

### Do we have automatic version tracking in GCS?
**Yes, we can!** But it depends entirely on how the GCS bucket is configured in GCP.

1. **If GCS Object Versioning is ENABLED (Recommended):**
   GCS natively supports Object Versioning. If this is turned on for `$GCS_BUCKET`, overwriting `raw_logs.zip` does *not* delete the old one. GCS simply archives the original zip as an older version (which you can view or restore via the GCP console or `gsutil`).
   In this scenario, our architecture is perfect: we get a single, clean "latest" URL for the UI to download, but GCP quietly maintains the history of all failed attempts in the background.

2. **If GCS Object Versioning is DISABLED:**
   The original `raw_logs.zip` from the failed attempt is permanently destroyed and replaced by the healed version. The BigQuery database will show that the first attempt failed, but the logs for *why* it failed (if they were only in that zip) will be lost, replaced by the logs of the successful attempt.

### Alternative Approach (Explicit Folder Separation)
If we absolutely cannot rely on GCS Object Versioning, we would need to change `worker/entrypoint.sh` to upload artifacts to a path defined by the `BATCH_JOB_ID` rather than the `BENCHHUB_RUN_ID`.
```bash
# Download from the original
gsutil cp "gs://$GCS_BUCKET/$RESUME_FROM_RUN_ID/raw_logs.zip" .

# ... Harbor runs ...

# Upload to a brand new folder using the Cloud Batch ID
DEST="$BUCKET_DIR/$BATCH_JOB_ID"
```
*   **Pros:** Explicit, guaranteed artifact history without relying on GCP bucket settings.
*   **Cons:** Fragmentation. The UI would need to know *which* `BATCH_JOB_ID` folder to look in when a user clicks "Download Logs" for a specific `run_id`. (Though, since BigQuery's `eval_results_jobs` table now tracks `job_id`, we could easily join this information!)

For now, the overwriting behavior paired with GCS Object Versioning is the cleanest approach as it keeps the UI's artifact fetching logic simple while preserving history at the infrastructure layer.
