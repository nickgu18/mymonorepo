## Harbor Native Resume & Artifact Versioning

The "Heal" feature relies on a unified identifier strategy to orchestrate repairs across Google Cloud Batch and Harbor seamlessly. By enforcing that Harbor's internal `eval_id` perfectly matches the overarching `BENCHHUB_RUN_ID`, the system can execute deterministic recovery workflows.

When a heal is requested, the worker intercepts the `RESUME_FROM_RUN_ID` environment variable. Instead of starting a fresh job, it downloads the existing artifacts from GCS, unpacks them into a directory matching the `eval_id`, and executes `harbor jobs resume`. Harbor natively detects which trials failed, selectively re-executes them, and merges the new successes into the existing `result.json`.

### GCS Artifact Versioning Recommendation
To maintain a pristine history of artifacts without complicating the frontend UI, the worker is designed to **overwrite** the `raw_logs.zip` in the original GCS directory (`gs://$GCS_BUCKET/$BENCHHUB_RUN_ID/raw_logs.zip`). 

To prevent the loss of historical logs from the failed attempts, **it is highly recommended to enable Object Versioning on the GCS bucket.**

*   **With Object Versioning Enabled:** When the healed `raw_logs.zip` is uploaded, GCS automatically archives the original failed zip as "v1" and serves the healed zip as the latest version. The UI only ever needs to fetch the single active link, but administrators retain full historical observability.
*   **Without Object Versioning:** The original logs for the failed tasks are permanently overwritten by the successful logs, severing the ability to post-mortem the original infrastructure failure.
